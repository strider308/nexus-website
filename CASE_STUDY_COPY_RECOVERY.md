# Case Study Copy Recovery & References

This file compiles the recovered copy models, taglines, problem narratives, designed-for audiences, and disclaimers from the pre-clean-sheet version of the website (specifically from tag `pre-clean-sheet-3d-rebuild` in `src/lib/content/nexus.ts` and related files).

## 1. ClinicOS Recovery
*   **Headline / Tagline:** "Run the patient journey from enquiry to pharmacy through one connected workflow."
*   **Operating Problem:** Many outpatient clinics lack a unified patient journey. Reception handles calls, walk-ins, and arrivals. Doctors handle vitals and consultation files. Cashiers handle payment, billing, and receipts. Pharmacy handles prescription checks and stock. Managers oversee logs. Every handoff causes delayed data, lost paper prescriptions, billing re-entry errors, and inventory mismatches.
*   **Designed For:** Independent outpatient clinics, family-owned polyclinics, multi-speciality chambers, doctor chambers, clinics with pharmacy counters, and growing clinic groups.
*   **Roles:** Patient & family, receptionist, doctor, nurse/assistant, diagnostic staff, billing/cashier, pharmacist, manager, owner/administrator.
*   **Workflow:** Enquiry -> Registration -> Appointment/Token -> Arrival -> Queue -> Consultation -> Diagnostic Order -> Billing -> Pharmacy -> Visit Completion -> Follow-up.
*   **Disclaimer:** Outpatient platform. Does not provide medical advice. Clinical decisions remain with qualified professionals. Queue times are estimates.

## 2. RestaurantOS Recovery
*   **Headline / Tagline:** "Connect every table, kitchen, service, and owner workflow."
*   **Operating Problem:** Operations are divided across paper tickets, verbal details, staff memory, billing counters, and owner spreadsheets. Cooks receive incomplete context, allergens are verbally repeated, waiters check prep status by walking to kitchens, cashiers tally bills manually, and owners reconcile POS collections after the shift.
*   **Designed For:** Table service restaurants, casual dining, multi-station kitchens, bistros, cafes, fine dining, and multi-location venues.
*   **Roles:** Guest, waiter/service staff, kitchen chef/staff, cashier, floor manager, owner.
*   **Workflow:** Menu access -> Order -> Kitchen ticket -> Preparation -> Ready -> Service -> Additional rounds -> Bill request -> Payment -> Shift reconciliation -> Owner review.
*   **Disclaimer:** Outcomes depend on adoption, setup, and connected systems. Tax and invoicing details must match local configurations.

## 3. ShipWright Recovery
*   **Headline / Tagline:** "Keep projects, tasks, files, decisions, blockers, and check-ins connected."
*   **Operating Problem:** Project context scatters across tools. The task is in a tracker, the discussion is in chat, files live in cloud drives, decisions are in doc logs, and blockers are mentioned once and forgotten. Managers spend hours reconstructing weekly status updates.
*   **Designed For:** Distributed engineering teams, remote product squads, early-stage startups, agencies, and cross-functional teams.
*   **Roles:** Operator/founder, product manager, engineering lead, developer, designer/contributor, project stakeholder.
*   **Workflow:** Project creation -> Milestone planning -> Task assignment -> Work discussion -> File & decision capture -> Check-in -> Blocker detection -> Dependency resolution -> Review -> Completion -> Project summary.
*   **Disclaimer:** Depends on team communication norms and consistency in logging context.

## 4. Supporting Systems (Aarogya, SecureScan, SafeDate, BuildPublic)
*   **Aarogya:** Metric tracker for BP, glucose, weight, and meal logs in Indian food contexts. Stores data locally. Does not diagnose or replace professional medical care.
*   **SecureScan:** Dev-focused package/script security check runner in CI/CD pipeline. Renders developer-friendly finding cards. Does not replace manual penetration testing.
*   **SafeDate:** Timed check-in safety planner routing SMS notifications to trusted contacts. Respects privacy; does not guarantee personal safety or replace emergency services.
*   **BuildPublic:** Founder execution board converting private check-ins into public roadmap feeds. Shields private configurations while compiling public updates.
